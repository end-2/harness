# 컨텍스트 맵 생성 프롬프트

## 입력

```
scan 에이전트 출력: {{scan_output}}
detect 에이전트 출력: {{detect_output}}
analyze 에이전트 출력: {{analyze_output}}
토큰 예산: {{token_budget}}
집중 분석 영역: {{focus_areas}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙에 따라 세 에이전트의 결과를 통합하여 4섹션 최종 산출물을 생성하세요. 사용자에게 질문하지 말고 결과만 출력합니다.

### Step 1: 토큰 예산 확인

`{{token_budget}}`가 지정되지 않으면 기본값 4000을 사용합니다. `{{focus_areas}}`가 지정된 경우, 해당 영역의 컴포넌트/기술에 더 많은 토큰을 할당합니다. 시스템 프롬프트 **"토큰 예산 관리 → 우선순위 기반 정보 선별"** 표에 따라 우선순위를 결정합니다.

### Step 2: 메타데이터 생성

시스템 프롬프트 **"메타데이터 생성"** 절의 스키마에 따라 산출물 최상단에 분석 메타데이터를 포함합니다.

### Step 3: 섹션 1 — 프로젝트 구조 맵 통합

`{{scan_output}}`을 시스템 프롬프트 **"4섹션 통합 매핑 → 섹션 1"** 스키마에 따라 변환합니다. `directory_tree`는 토큰 예산에 맞춰 추가 축약할 수 있습니다.

### Step 4: 섹션 2 — 기술 스택 탐지 통합

`{{detect_output}}`의 `tech_stack`을 시스템 프롬프트 **"4섹션 통합 매핑 → 섹션 2"** 스키마에 따라 포함합니다. 토큰 예산이 부족하면 시스템 프롬프트 **"축약 전략 → 외부 의존성 요약"** 지침에 따라 보조 도구(린터, 포매터)를 카테고리별로 요약합니다. `tech_relationships`도 함께 포함합니다.

### Step 5: 섹션 3 — 컴포넌트 관계 분석 통합

`{{analyze_output}}`의 `components`를 시스템 프롬프트 **"4섹션 통합 매핑 → 섹션 3"** 스키마에 따라 포함합니다. 시스템 프롬프트 **"축약 전략 → 계층적 상세"**에 따라 핵심 컴포넌트는 전체 필드, 유틸/설정 컴포넌트는 이름+역할만 기록합니다.

### Step 6: 섹션 4 — 아키텍처 추론 통합

`{{analyze_output}}`의 `architecture_inference`에 `{{detect_output}}`의 테스트/빌드 정보를 보강하여, 시스템 프롬프트 **"4섹션 통합 매핑 → 섹션 4"** 스키마에 따라 출력합니다. `test_patterns`, `build_deploy_patterns`, `token_budget_summary`를 포함합니다.

### Step 7: 일관성 검증

시스템 프롬프트 **"일관성 검증"** 절의 4개 항목(ID 일관성, 경로 일관성, 기술-컴포넌트 매핑, 진입점-컴포넌트 매핑)을 확인합니다. 불일치 발견 시 수정하고, 수정 내용을 metadata에 기록합니다.

### Step 8: 토큰 예산 최종 조정

예산 초과 시 시스템 프롬프트 **"토큰 예산 관리 → 우선순위 기반 정보 선별"** 표의 우선순위 역순으로 축약합니다:

1. 개별 파일 목록 → 카운트로 대체
2. 디렉토리 트리 상세 → 상위 2단계로 축약
3. 외부 의존성 목록 → 카테고리별 요약
4. 유틸/설정 컴포넌트 → 그룹 요약
5. 횡단 관심사 → 상위 3개만

**절대 삭제하지 않는 정보**: 진입점, API 표면, 아키텍처 스타일, 주요 기술(언어/프레임워크/DB).

### Step 9: 산출물 출력

시스템 프롬프트 **"4섹션 통합 매핑"** 전체 스키마에 맞춰 YAML로 출력하고, **"출력 프로토콜"**에 따라 `meta.json`/`body.md`에 기록합니다.
