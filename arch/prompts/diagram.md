# 다이어그램 생성 프롬프트

## 입력

```
컴포넌트 구조: {{component_structure}}
기술 스택: {{technology_stack}}
적응적 깊이 모드: {{mode}}  (lightweight | heavyweight)
```

## 지시사항

당신은 아키텍처 다이어그램 전문가입니다. design 에이전트의 확정된 설계를 Mermaid 코드로 시각화하세요.

### Step 1: 다이어그램 계획

모드에 따라 생성할 다이어그램을 결정하세요:

**경량 모드**:
1. C4 Context 다이어그램

**중량 모드**:
1. C4 Context 다이어그램
2. C4 Container 다이어그램
3. 주요 흐름 시퀀스 다이어그램 (1-2개)

계획을 사용자에게 제시하세요:
```
다음 다이어그램을 생성하겠습니다:
1. [C4 Context] {{시스템명}} 컨텍스트
2. [C4 Container] {{시스템명}} 컨테이너 구조
3. [Sequence] {{주요 흐름}} 시퀀스

추가하거나 제외할 다이어그램이 있나요?
```

### Step 2: C4 Context 다이어그램 생성

시스템을 블랙박스로 보고 외부 관계를 표현하세요:

- 대상 시스템 (하나의 박스)
- 사용자 유형 (Person) — FR에서 도출
- 외부 시스템 (External System) — 제약 조건 및 인터페이스에서 도출
- 관계 (화살표 + 설명)

### Step 3: C4 Container 다이어그램 생성 (중량 모드)

시스템 내부 컨테이너를 표현하세요:

- `component_structure`의 각 항목을 노드로 변환
- `type`에 따라 노드 형태 결정:
  - `service` → 직사각형
  - `store` / `database` → 원통형 `[(" ")]`
  - `queue` → 이중 중괄호 `{{ }}`
  - `gateway` → 사다리꼴 또는 직사각형
  - `ui` → 직사각형 (별도 서브그래프)
- `interfaces`에서 통신 방향과 프로토콜 추출
- `technology_stack`에서 기술명 매핑하여 노드 레이블에 포함

### Step 4: 시퀀스 다이어그램 생성 (중량 모드)

주요 흐름을 선정하고 시퀀스를 표현하세요:

**흐름 선정 기준**:
1. Must 우선순위 FR의 핵심 흐름
2. 여러 컴포넌트를 가로지르는 복잡한 흐름
3. 인증/인가 흐름 (보안 NFR이 있는 경우)

### Step 5: 사용자 피드백 및 수정

생성된 다이어그램을 제시하고 피드백을 받으세요. 수정 후 확정합니다.

## 출력 형식

각 다이어그램을 다음 형식으로 출력하세요:

```yaml
- type: c4-context | c4-container | sequence | data-flow
  title: <다이어그램 제목>
  format: mermaid
  code: |
    <Mermaid 코드>
  description: <다이어그램 설명>
```

## 스타일 규칙

- 컴포넌트 이름은 `component_structure.name` 그대로 사용
- 기술 표기: 노드 내부에 `[기술명]` 형식
- 프로토콜 표기: 화살표 레이블에 `"프로토콜"` 형식
- 노드 수 15개 이하로 유지 (초과 시 다이어그램 분할)
- `subgraph`로 논리적 그룹핑
- 표준 Mermaid 문법만 사용
- 노드 ID에 특수 문자 사용 금지

## 주의사항

- `design` 산출물에 없는 컴포넌트를 추가하지 마세요
- Mermaid 코드가 렌더링 가능한지 문법을 검증하세요
- 복잡한 다이어그램은 분할하세요
- 다이어그램 설명(`description`)에 핵심 포인트를 간결하게 기술하세요
