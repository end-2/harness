# ADR-001: 모듈러 모놀리스 아키텍처 채택

## 상태

accepted

## 컨텍스트

휴가 관리 시스템의 전체 아키텍처 스타일을 결정해야 한다. 시스템은 인증, 휴가 신청/승인, 알림, 관리 기능을 포함하며, 4명의 개발팀이 3개월 내에 1차 릴리스를 완료해야 한다.

**RE 근거**:
- CON-002: 사용자 규모 500명, 동시 접속 최대 100명 — 대규모 분산 처리 불필요
- CON-004: 3개월 내 1차 릴리스 — 개발/운영 오버헤드 최소화 필요
- QA:availability: 업무 시간 99.5% 가용성 — 단일 서버로 달성 가능한 수준
- QA:maintainability: 휴가 정책 변경 용이성 — 모듈 경계로 변경 영향 범위 제한

## 결정

시스템을 **모듈러 모놀리스**로 구성한다. 단일 배포 단위 내에서 도메인 모듈(인증, 휴가관리, 알림, 관리)을 명확히 분리하고, 모듈 간 인터페이스를 정의하여 향후 필요 시 서비스 분리가 가능하도록 한다.

## 대안 비교

| 기준 | 모듈러 모놀리스 | 마이크로서비스 | 전통적 모놀리스 |
|------|---------------|--------------|---------------|
| 개발 속도 | 빠름 (단일 코드베이스) | 느림 (서비스 간 통합) | 빠름 |
| 운영 복잡도 | 낮음 (단일 배포) | 높음 (K8s 필요) | 낮음 |
| 팀 역량 적합성 | 높음 (Django 경험) | 낮음 (K8s 미경험) | 높음 |
| 3개월 일정 달성 | 가능 | 리스크 높음 | 가능 |
| 향후 확장성 | 모듈 분리 가능 | 독립 확장 가능 | 분리 어려움 |
| 500명 규모 대응 | 충분 | 과도함 | 충분 |
| **결론** | **선택** | 기각 | 기각 |

## 결과

### 긍정적 결과
- 4명 팀이 단일 코드베이스에서 효율적으로 협업 가능
- 배포/운영 복잡도 최소화로 3개월 일정 달성 가능성 높음
- 모듈 경계를 통해 코드 변경의 영향 범위 제한
- Django의 앱(app) 구조를 활용하여 자연스럽게 모듈 분리

### 부정적 결과 / 트레이드오프
- 개별 모듈의 독립 배포/확장 불가 — 500명 규모에서는 문제 없으나 대규모 성장 시 리스크
- 모듈 간 인터페이스 준수는 개발자 규율에 의존 — 코드 리뷰로 보완 필요
- 단일 프로세스 장애 시 전체 서비스 중단 — Docker restart 정책으로 대응

### 후속 조치
- Django 앱 구조를 활용한 모듈 분리 가이드라인 수립
- 모듈 간 직접 import 대신 인터페이스를 통한 호출 규칙 정립
- 2차 릴리스에서 사용자 증가 시 서비스 분리 검토

## 관련 문서
- RE: CON-002, CON-004, QA:availability, QA:maintainability
- 관련 ADR: ADR-002 (relates-to — 프론트엔드/백엔드 구성은 이 결정의 하위 결정)

# ADR-002: SPA + REST API 아키텍처

## 상태

accepted

## 컨텍스트

프론트엔드 아키텍처를 결정해야 한다. 사용성(usability)이 최우선 품질 속성(priority 1)이며, 사용자가 빠르고 직관적으로 휴가를 신청할 수 있어야 한다. 팀 캘린더(FR-004) 등 인터랙티브한 UI가 필요하다.

**RE 근거**:
- QA:usability (priority 1): "신규 사용자가 5분 이내 첫 휴가 신청 완료"
- NFR-001: 초기 페이지 로드 3초 이내, 이후 전환 500ms 이내
- FR-004: 팀 휴가 캘린더 (월별/주별 전환, 3초 이내 렌더링)

## 결정

프론트엔드는 **React SPA**, 백엔드는 **Django REST API**로 구성한다. 프론트엔드와 백엔드를 별도 프로젝트로 분리하되, 동일 서버에 Nginx를 통해 배포한다.

## 대안 비교

| 기준 | React SPA + Django REST | Django Template (SSR) | Next.js + Django |
|------|------------------------|----------------------|------------------|
| 페이지 전환 속도 | 빠름 (CSR) | 느림 (매번 서버 렌더링) | 빠름 (SSR+CSR) |
| 사용성(usability) | 높음 | 보통 | 높음 |
| 팀 경험 | React 2년+ | Django 3년+ | Next.js 미경험 |
| 개발 복잡도 | 보통 (API 분리) | 낮음 (일체형) | 높음 (프레임워크 학습) |
| SEO | 불필요 (사내 시스템) | 유리 | 유리 |
| 3개월 일정 | 가능 | 가능 | 리스크 |
| **결론** | **선택** | 기각 | 기각 |

## 결과

### 긍정적 결과
- SPA의 빠른 페이지 전환으로 usability(priority 1) 충족
- 팀 기존 React + Django 역량 활용
- API 분리로 향후 모바일 앱 등 다른 클라이언트 지원 가능

### 부정적 결과 / 트레이드오프
- 초기 번들 로딩이 SSR 대비 느림 (3초 이내 목표로 관리)
- 프론트엔드/백엔드 분리로 CORS, 인증 토큰 관리 등 추가 복잡도

### 후속 조치
- SPA 번들 최적화 (코드 스플리팅, 레이지 로딩)으로 초기 로딩 3초 이내 달성
- SAML 인증 후 JWT 토큰 발급 방식으로 SPA-API 인증 처리

## 관련 문서
- RE: QA:usability, NFR-001, FR-004
- 관련 ADR: ADR-001 (relates-to — 모듈러 모놀리스 내 프론트엔드/백엔드 구성)

# ADR-003: PostgreSQL 단일 데이터베이스

## 상태

accepted

## 컨텍스트

데이터 저장 전략을 결정해야 한다. 프라이빗 클라우드(CON-003)에서 직접 운영해야 하므로 팀이 운영 가능한 기술을 선택해야 한다. 사용자 규모는 500명(CON-002)이다.

**RE 근거**:
- CON-003 (hard): 프라이빗 클라우드 배포 — 관리형 DB 서비스 불가, 직접 운영 필수
- CON-002: 사용자 규모 500명, 동시 접속 100명
- NFR-001: API 응답 P95 1초 이내

## 결정

**PostgreSQL 14**를 단일 인스턴스로 사용하고, 도메인 모듈별로 스키마(schema)를 분리하여 논리적 경계를 유지한다.

## 대안 비교

| 기준 | PostgreSQL 단일 | PostgreSQL + Redis 캐시 | MySQL |
|------|----------------|----------------------|-------|
| 팀 운영 경험 | 있음 | PostgreSQL 있음, Redis 일부 | 없음 |
| 운영 복잡도 | 낮음 (1개 서비스) | 보통 (2개 서비스) | 낮음 |
| 100명 동시 접속 대응 | 충분 (인덱스 최적화) | 충분+ (캐시 히트) | 충분 |
| P95 1초 달성 | 가능 | 확실 | 가능 |
| 프라이빗 클라우드 운영 | 가능 | 가능 | 가능 |
| **결론** | **선택** | 기각 | 기각 |

## 결과

### 긍정적 결과
- 운영 포인트 최소화 (단일 DB)
- 팀 기존 PostgreSQL 운영 경험 활용
- 스키마 분리로 모듈 경계 유지 (ADR-001과 일관)

### 부정적 결과 / 트레이드오프
- 캐시 레이어 없이 모든 읽기가 DB 직접 접근 — 100명 수준에서는 문제 없음
- 단일 인스턴스로 SPOF — 정기 백업과 복구 절차로 대응
- 수평 확장 불가 — 수직 확장으로 700명까지 대응, 이후 읽기 복제 검토

### 후속 조치
- 주요 쿼리에 대한 인덱스 최적화
- 정기 백업 스크립트(pg_dump) 구성
- 2차 릴리스에서 스트리밍 복제 검토

## 관련 문서
- RE: CON-002, CON-003, NFR-001
- 관련 ADR: ADR-001 (relates-to — 모듈러 모놀리스의 데이터 전략)
