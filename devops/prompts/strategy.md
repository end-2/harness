# 배포 전략 프롬프트

## 입력

```
SLO 산출물: {{slo_output}}
Arch 산출물: {{arch_output}}
Pipeline 산출물: {{pipeline_output}}
```

## 지시사항

당신은 배포 전략 설계 전문가입니다. SLO 목표와 Arch 결정을 분석하여 최적의 배포 방식과 롤백 절차를 자동 결정하세요.

### Step 1: SLO 분석

SLO 산출물에서:

- 가용성 SLO 목표치를 확인
- 현재 에러 버짓 잔량을 확인 (초기: 100%)
- 에러 버짓 정책을 확인

### Step 2: 배포 방식 결정

의사결정 트리를 따라 배포 방식을 자동 선택하세요:

```
가용성 목표 ≥ 99.99% → 블루/그린 + 카나리
가용성 목표 ≥ 99.9%  → 에러 버짓 < 20%? → 카나리 (느린 롤아웃)
                      → 에러 버짓 ≥ 20%? → 블루/그린
가용성 목표 ≥ 99.5%  → 롤링 업데이트
가용성 목표 < 99.5%  → 롤링 업데이트 또는 Recreate
```

선택 근거를 `rationale`에 기록하세요.

### Step 3: 배포 순서 결정

Arch `component_structure.dependencies`에서:

1. 데이터베이스 마이그레이션 → 최우선
2. 공유 라이브러리/인프라 → 의존하는 서비스보다 먼저
3. 서비스 → 의존성 역순 (leaf → root)
4. 게이트웨이 → 모든 서비스 배포 후 마지막

### Step 4: 롤백 트리거 설정

SLO 번-레이트 알림과 연결:

- 빠른 소진 (14.4x, 1h): 자동 롤백 트리거
- 헬스 체크 연속 실패 (3회): 자동 롤백 트리거
- 에러율 급증 (5xx > 5%, 5분): 자동 롤백 트리거

### Step 5: 롤백 절차

배포 방식별 롤백 절차를 설계하세요:

- **블루/그린**: 이전 환경(블루)으로 트래픽 전환
- **카나리**: 카나리 트래픽 차단, 이전 버전 유지
- **롤링**: `kubectl rollout undo` 또는 이전 태그 재배포
- **Recreate**: 이전 버전 재배포 (다운타임 발생)

### Step 6: 헬스 체크 정의

Arch `component_structure.interfaces`에서 헬스 체크 엔드포인트를 도출:

- Liveness: 프로세스 생존 확인 (`/healthz`)
- Readiness: 트래픽 수신 가능 확인 (`/ready`)
- Startup: 초기화 완료 확인

### Step 7: Pipeline 역반영

결정된 배포 전략을 Pipeline 산출물의 배포 스테이지에 반영하세요:

- 배포 방식에 따른 배포 명령어 추가
- 롤백 트리거 조건을 배포 후 검증 스텝에 포함
- 헬스 체크를 배포 후 검증에 포함

### Step 8: 산출물 정리

다음 형식으로 산출물을 정리하세요:

**배포 전략 요약**: 배포 방식, 롤아웃 단계, 롤백 방식, 배포 순서, 근거
**롤백 절차**: 단계, 조치, 담당, 예상 소요
**헬스 체크**: 대상 컴포넌트, 체크 유형, 경로, 간격, 실패 임계
**Pipeline 역반영**: 변경된 배포 스테이지 설정

## 주의사항

- SLO 목표를 임의로 변경하지 마세요. SLO는 RE 품질 속성에서 도출된 것입니다
- 배포 방식과 인프라 제약(비용, 리소스)이 충돌하면 에스컬레이션하세요
- 데이터베이스 마이그레이션은 항상 서비스 배포보다 먼저 실행되어야 합니다
- 롤백 트리거는 SLO 번-레이트 알림과 연결되어야 합니다 (원시 임계값이 아닌 번-레이트 기반)
