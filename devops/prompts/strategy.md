# 배포 전략 프롬프트

## 입력

```
SLO 산출물: {{slo_output}}
Arch 산출물: {{arch_output}}
Pipeline 산출물: {{pipeline_output}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙을 따라 SLO 목표와 Arch 결정을 분석하여 최적의 배포 방식과 롤백 절차를 자동 결정하세요.

### Step 1: SLO 분석

SLO 산출물에서 가용성 SLO 목표치, 현재 에러 버짓 잔량(초기 100%), 에러 버짓 정책을 확인합니다.

### Step 2: 배포 방식 결정

시스템 프롬프트 **"SLO + Arch → 배포 전략 의사결정 트리"**를 따라 배포 방식을 자동 선택하고, 시스템 프롬프트 **"배포 방식별 상세"**(블루/그린, 카나리, 롤링, Recreate)에서 선택된 방식의 상세 파라미터를 적용합니다. 선택 근거를 `rationale`에 기록합니다.

### Step 3: 배포 순서 결정

시스템 프롬프트 **"배포 순서 결정"** 절의 우선순위(DB 마이그레이션 → 공유 인프라 → 서비스 의존성 역순 → 게이트웨이)를 Arch `component_structure.dependencies`에 적용합니다.

### Step 4: 롤백 트리거 설정

시스템 프롬프트 **"자동 롤백 트리거"** 표(SLO 번-레이트 / 헬스 체크 / 에러율)에 따라 트리거를 SLO 번-레이트 알림과 연결합니다.

### Step 5: 롤백 절차

시스템 프롬프트 **"롤백 절차"** 절의 5단계(트래픽 차단 → 이전 버전 복원 → 헬스 체크 → 알림 → 사후 분석 트리거)를 선택된 배포 방식에 맞게 구체화합니다.

### Step 6: 헬스 체크 정의

Arch `component_structure.interfaces`에서 엔드포인트를 도출하여 시스템 프롬프트 **"헬스 체크 정의"** 표(Liveness / Readiness / Startup)의 파라미터를 적용합니다.

### Step 7: Pipeline 역반영

결정된 배포 전략을 Pipeline 산출물의 배포 스테이지에 반영합니다:

- 배포 방식에 따른 배포 명령어 추가
- 롤백 트리거 조건을 배포 후 검증 스텝에 포함
- 헬스 체크를 배포 후 검증에 포함

시스템 프롬프트 **"트래픽 관리"** 절(가중치/헤더/지역 기반)의 전략도 필요 시 포함합니다.

### Step 8: 산출물 정리

시스템 프롬프트 **"출력 형식"** 표(배포 전략 요약 / 롤백 절차 / 헬스 체크)에 맞춰 정리하고, **"출력 프로토콜"**에 따라 `meta.json`/`body.md`에 기록합니다. 배포 전략-인프라 충돌 시 시스템 프롬프트 **"에스컬레이션 조건"**을 따릅니다.
