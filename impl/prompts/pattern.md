# 디자인 패턴 프롬프트

## 입력

```
문제 상황/코드: {{problem_context}}
아키텍처 결정: {{architecture_decisions}}
컴포넌트 구조: {{component_structure}}
기술 스택: {{technology_stack}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙을 따라 문제 상황을 분석하고 적절한 디자인 패턴을 추천/적용하세요. 에스컬레이션하지 않습니다.

### Step 1: Arch 명시 패턴 확인

`architecture_decisions`에서 명시적으로 요구하는 패턴을 추출하고, 시스템 프롬프트 **"패턴 적용 의사결정 → Arch 명시 패턴 (필수 적용)"** 절차에 따라 필수 적용 대상을 확정합니다 (skip 불가).

### Step 2: 문제 상황 분석

`problem_context`를 분석하여 패턴 적용이 유의미한 지점을 식별합니다. 시스템 프롬프트 **"패턴 의사결정 트리"**(생성/구조/행위 패턴)를 순서대로 적용하여 후보 패턴을 선정합니다.

### Step 3: 과도한 패턴 적용 경계

시스템 프롬프트 **"패턴 적용 체크리스트"**를 통해 후보 패턴의 적용 여부를 판단합니다. YAGNI 원칙에 따라 조건 분기가 2개 이하거나 확장 가능성이 낮으면 적용하지 않습니다. 기술 스택의 관용적 구현이 있으면 그것을 우선합니다.

### Step 4: 패턴 적용

선정된 패턴을 적용하고 전후 비교를 제시합니다:

```
[패턴: <이름>]
문제 상황: <설명>
적용 근거: <AD-xxx (필수) 또는 문제 상황 근거>

적용 전:
  <코드>

적용 후:
  <코드>

트레이드오프:
  + <장점>
  - <단점>
```

### Step 5: 안티패턴 탐지

시스템 프롬프트 **"안티패턴 탐지"** 표의 증상과 교정 방법에 따라 코드에서 안티패턴을 탐지하고 교정 방안을 제시합니다.

### Step 6: 구현 결정 기록 (IDR)

모든 패턴 적용(Arch 명시 + 자체 판단)에 대해 IDR을 작성합니다. 미적용 결정은 기록하지 않습니다.

```yaml
- id: IDR-xxx
  title: "[패턴명]을 [대상]에 적용"
  decision: "[구체적 적용 방법]"
  rationale: "[Arch 결정 참조 또는 문제 상황 설명]"
  alternatives_considered:
    - "[대안 1] — [기각 사유]"
    - "[대안 2] — [기각 사유]"
  pattern_applied: "[패턴명]"
  arch_refs: ["AD-xxx"]
  re_refs: ["FR-xxx"]
```

### Step 7: 산출물 출력

시스템 프롬프트 **"출력 형식"** 및 **"출력 프로토콜"**에 따라 `recommended_patterns`와 `implementation_decisions`를 `meta.json`/`body.md`에 기록합니다.
