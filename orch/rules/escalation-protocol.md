# 에스컬레이션 규약 (Escalation Protocol)

스킬 에이전트가 사용자 개입을 요청하는 조건과 방법을 정의합니다.

---

## 1. 스킬 유형별 에스컬레이션 정책

### 대화형 스킬 (Dialogue Skills)

의미 있는 질문 묶음마다 `needs_user_input`을 사용합니다.

| 스킬:에이전트 | 일반적인 에스컬레이션 상황 |
|--------------|------------------------|
| re:elicit | 비즈니스 도메인 이해, 우선순위 확인, 기능 범위 조율 |
| re:analyze | 요구사항 충돌 해결, 우선순위 판단 |
| re:spec | 명세 초안 확인, 누락 사항 보완 |
| re:review | 리뷰 결과 확인, 변경 승인 |
| arch:design | 기술 스택 선택, 아키텍처 트레이드오프 결정 |
| sec:threat-model | 자산 가치 평가, 보안 우선순위 확인 |

### 자동 실행 스킬 (Auto-execute Skills)

예외 조건에서만 `needs_user_input`을 사용합니다. 정상 실행 시 사용자 개입 없이 완료됩니다.

| 스킬:에이전트 | 에스컬레이션 조건 (예외만) |
|--------------|------------------------|
| ex:scan | 프로젝트 루트 접근 불가, 심볼릭 링크 순환 |
| ex:detect | 매니페스트 파일 전무 |
| ex:analyze | 순환 의존성으로 분석 불가 |
| ex:map | 토큰 예산 초과로 정보 손실 불가피 |
| impl:generate | 아키텍처 결정이 코드로 실현 불가능 |
| qa:generate | 테스트 대상이 명확하지 않음 |
| sec:audit | 치명적 취약점 발견 시 즉시 보고 |
| sec:review | 보안 정책 위반 발견 |
| sec:compliance | 규정 준수 여부 판단 불가 |
| devops:* | 인프라 제약으로 구성 불가 |

## 2. 에스컬레이션 필수 포함 정보

모든 `needs_user_input` 신호에 다음 정보를 포함합니다:

```markdown
## needs_user_input
- skill: "<스킬:에이전트>"
- turn: <대화 턴 번호>
- questions:
    - id: "<질문 ID>"
      text: "<질문 텍스트>"
      context: "<왜 이 정보가 필요한지 설명>"
      type: <open | choice | confirmation>
      options: [<선택지 목록 — choice 유형만>]
      default: "<기본값 — 있는 경우>"
- partial_output: { <지금까지 생성된 부분 산출물> }
```

### 질문 유형

| 유형 | 설명 | 사용 상황 |
|------|------|----------|
| open | 자유 텍스트 응답 | 도메인 지식, 구체적 수치, 설명 필요 |
| choice | 선택지 중 택일 | 기술 스택, 아키텍처 패턴, 전략 선택 |
| confirmation | 예/아니오 확인 | 초안 승인, 방향 확인, 위험 수용 |

## 3. 에스컬레이션 원칙

1. **질문 묶기**: 관련된 질문은 하나의 `needs_user_input`에 묶어서 전달합니다. 질문 하나마다 에스컬레이션하지 마세요
2. **맥락 제공**: 각 질문에 `context`를 포함하여 사용자가 왜 이 정보가 필요한지 이해할 수 있게 합니다
3. **기본값 제안**: 가능하면 `default` 값을 제안하여 사용자가 빠르게 진행할 수 있게 합니다
4. **부분 산출물 포함**: 에스컬레이션 시점까지 생성된 부분 결과를 `partial_output`에 포함합니다
5. **코드 우선**: 코드나 명세에서 답을 찾을 수 있다면 에스컬레이션하지 마세요
