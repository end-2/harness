# 요구사항 분석 출력 예시

> 휴가 관리 시스템 요구사항 분석 결과

## 요약

전체 9개 요구사항, 5개 제약 조건을 분석했습니다.

- 충돌: 1건 (자원 충돌)
- 누락: 4건 (Major 2건, Minor 2건)
- 위험: 2건
- 사용자 의사결정 필요: 5건

---

## 충돌 사항

| 요구사항 A | 요구사항 B | 충돌 유형 | 설명 | 해결 옵션 |
|-----------|-----------|----------|------|----------|
| NFR-001 (페이지 로드 2초) | CON-003 (OpenStack 배포) | 자원 충돌 | 프라이빗 클라우드의 CDN 부재로 정적 자원 로딩이 느릴 수 있음 | A: 사내 CDN/캐시 서버 구축 [권고]<br>B: 성능 목표를 3초로 완화<br>C: SPA로 구현하여 초기 로딩 후 빠른 네비게이션 |

## 누락 사항

| 영역 | 누락 내용 | 심각도 | 권고 |
|------|----------|--------|------|
| 기능 | 휴가 신청 취소/수정 기능 | Major | FR-002에 취소/수정 시나리오 추가 필요 |
| 기능 | 대리 승인 (팀장 부재 시) | Major | 별도 FR로 추가 또는 의도적 제외 확인 필요 |
| 비기능 | 감사 로그 (누가 언제 승인/반려했는지) | Minor | 개인정보보호법 준수(CON-005)를 위해 권고 |
| 비기능 | 백업/복구 정책 | Minor | 사내 클라우드 환경에서의 데이터 보호 전략 필요 |

## 위험 요소

| 요구사항 | 위험 유형 | 설명 | 완화 방안 |
|---------|----------|------|----------|
| CON-004 (3개월) | 일정 | Must 요구사항 5개 + Should 4개를 3개월 내 구현해야 함. Should 항목 일부를 2차 릴리스로 분리 필요 | MoSCoW 기반 2단계 릴리스 계획 수립 |
| CON-005 (개인정보) | 규제 | 병가 사유의 개인정보 처리 기준이 불명확. 수집 동의, 보관 기간, 접근 권한 정의 필요 | 개인정보 처리 방침 수립 후 관련 FR/NFR 추가 |

## 의존 관계

```
FR-001 (SSO 로그인)
  └── FR-002 (휴가 신청) ← 인증 필요
      ├── FR-003 (승인/반려) ← 신청 존재 필요
      └── FR-005 (잔여 조회) ← 신청 데이터 필요
  └── FR-004 (캘린더) ← 인증 필요, FR-002 데이터 참조
  └── FR-007 (관리자) ← 인증 + 권한 필요

FR-006 (알림) ← FR-003에 의존 (승인/반려 이벤트)
```

## 품질 속성 트레이드오프

### usability vs security

| 옵션 | 설명 |
|------|------|
| A: 사용성 우선 [권고] | 민감 정보(병가 사유)는 기본 숨김, 권한 있는 사용자만 클릭하여 확인. UX 흐름을 단순하게 유지 |
| B: 보안 우선 | 민감 정보 접근 시 매번 재인증(2FA). 보안은 강화되나 사용성 저하 |

### maintainability vs 일정(CON-004)

| 옵션 | 설명 |
|------|------|
| A: 유지보수성 투자 | 정책 엔진을 별도 모듈로 설계. 초기 개발 기간 증가(+2주)하나 향후 정책 변경 용이 |
| B: 빠른 구현 [권고] | 1차 릴리스에서는 하드코딩, 2차에서 설정화. 3개월 일정 준수 가능 |

## 정제된 요구사항 목록

기존 요구사항에 다음을 추가/수정 제안:

- **FR-008** (추가 제안): 휴가 신청 취소/수정 — 대기중 상태에서만 취소/수정 가능
- **FR-009** (추가 제안): 대리 승인 — 팀장이 부재 시 지정된 대리인이 승인 가능
- **NFR-003** (추가 제안): 감사 로그 — 모든 신청/승인/반려/수정/취소에 대한 감사 로그 기록
