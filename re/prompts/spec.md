# 요구사항 명세 프롬프트

## 입력

```
정제된 요구사항: {{refined_requirements}}
검증된 제약 조건: {{validated_constraints}}
품질 속성 트레이드오프: {{quality_tradeoffs}}
사용자 의사결정 결과: {{user_decisions}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙을 따라 분석 완료된 요구사항을 세 섹션으로 구조화하세요.

### Step 1: 명세 모드 판별

시스템 프롬프트 **"적응적 명세 수준"** 의 경량/중량 모드 적용 조건으로 모드를 판별하고 사용자에게 확인받습니다:

```
분석 결과, 요구사항 {{count}}개, 제약 조건 {{count}}개로 [경량/중량] 모드가 적합합니다.
다른 모드를 원하시면 말씀해주세요.
```

### Step 2: 요구사항 명세 작성 (섹션 1)

시스템 프롬프트 **"섹션 1: 요구사항 명세"** 의 YAML 스키마에 맞춰 각 FR/NFR을 구조화합니다.

**체크리스트**:
- [ ] 모든 요구사항에 고유 ID가 부여되었는가 (ID 체계 준수)
- [ ] acceptance_criteria가 검증 가능한가 (정량적 기준 포함)
- [ ] priority가 MoSCoW 중 하나인가
- [ ] dependencies가 유효한 ID를 참조하는가
- [ ] source가 명확한가

### Step 3: 제약 조건 작성 (섹션 2)

시스템 프롬프트 **"섹션 2: 제약 조건"** 의 YAML 스키마에 맞춰 각 제약을 구조화합니다.

**체크리스트**:
- [ ] 모든 제약에 유형이 분류되었는가
- [ ] rationale이 충분한가
- [ ] flexibility가 적절한가
- [ ] 요구사항 구현을 불가능하게 하는 제약은 없는가

### Step 4: 품질 속성 우선순위 작성 (섹션 3)

시스템 프롬프트 **"섹션 3: 품질 속성 우선순위"** 의 YAML 스키마에 맞춰 각 속성을 구조화합니다.

**체크리스트**:
- [ ] metric이 정량적이고 측정 가능한가
- [ ] 우선순위 간 동률이 없는가 (동률 시 사용자에게 질문)
- [ ] trade_off_notes가 실질적인 트레이드오프를 설명하는가
- [ ] NFR의 metric과 일관성이 있는가

### Step 5: 초안 리뷰 요청

세 섹션의 초안을 사용자에게 제시하고 피드백을 요청합니다:

```
[섹션 1: 요구사항 명세]
...

[섹션 2: 제약 조건]
...

[섹션 3: 품질 속성 우선순위]
...

확인해주실 사항:
1. 빠지거나 잘못된 요구사항이 있나요?
2. 우선순위(MoSCoW)가 적절한가요?
3. 수용 기준이 충분히 구체적인가요?
4. 제약 조건의 유연성(hard/soft/negotiable) 분류가 맞나요?
5. 품질 속성 우선순위 순서에 동의하시나요?
```

### Step 6: 피드백 반영 및 최종 확인

사용자 피드백을 반영하여 수정한 후 최종 확인을 요청합니다. 시스템 프롬프트 **"후속 스킬 소비 계약"** 표를 참조하여 각 섹션이 소비자 스킬 요구를 만족하는지 점검합니다.

### Step 7: 결과 기록

시스템 프롬프트 **"출력 프로토콜"**에 따라 `meta.json`/`body.md`에 기록합니다. ID 체계(FR-XXX, NFR-XXX, CON-XXX)를 준수하고 한 번 부여된 ID는 변경하지 않습니다.

## 주의사항

- 후속 스킬이 파싱 없이 소비할 수 있도록 일관된 형식을 유지하세요
- 경량 모드에서도 세 섹션 구조는 유지합니다 (내용의 깊이만 조절)
- 기술 결정(아키텍처, 프레임워크 선택)은 제약 조건으로만 기록합니다. 설계 결정은 arch 스킬의 영역입니다
