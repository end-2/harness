# 요구사항 분석 프롬프트

## 입력

```
요구사항 후보: {{requirements_candidates}}
제약 조건 후보: {{constraints_candidates}}
품질 속성 힌트: {{quality_attribute_hints}}
미해결 질문: {{open_questions}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙을 따라 도출된 요구사항의 완전성·일관성·실현 가능성을 분석하고, 사용자 의사결정이 필요한 이슈를 제시하세요.

### Step 1: 입력 검수

elicit 산출물의 각 항목을 순회하며 형식적 완전성(ID, 제목, 설명, 출처 등)을 확인합니다.

### Step 2: 충돌 분석

시스템 프롬프트 **"충돌 및 모순 탐지"** 기준(직접/자원/암묵적 충돌)에 따라 모든 요구사항 쌍을 검토합니다. 각 충돌에 대해 충돌 쌍·유형·이유를 명시하고 2-3개의 해결 옵션을 제시하되 하나에 [권고] 태그를 부여합니다.

### Step 3: 누락 분석

시스템 프롬프트 **"누락 요구사항 식별"** 체크리스트(경계 조건, 예외 시나리오, 비기능 영역, 관리 기능, 데이터 관련)에 따라 빠진 영역을 탐지합니다.

### Step 4: 실현 가능성 평가

시스템 프롬프트 **"실현 가능성 평가"** 표의 세 축(기술적/시간적/비용적)으로 각 요구사항을 평가합니다. 위험도가 높은 요구사항은 별도로 표시합니다.

### Step 5: 의존 관계 분석

시스템 프롬프트 **"의존 관계 분석"** 기준(선행/병행/선택 의존)으로 요구사항 간 의존 그래프를 생성합니다.

### Step 6: 품질 속성 트레이드오프 분석

시스템 프롬프트 **"트레이드오프 제시"** 형식에 따라 품질 속성 힌트 간 상충 관계를 분석하고, 각 옵션의 장단점을 설명한 뒤 사용자에게 우선순위를 질문합니다.

### Step 7: 질문 생성

발견된 이슈마다 사용자 의사결정을 위한 질문을 생성합니다:

```
[Q-001] <이슈 요약>
  옵션 A: ... [권고]
  옵션 B: ...
  옵션 C: ...
  → 어떤 옵션을 선호하시나요?
```

미해결 질문(elicit 단계에서 넘어온 항목)도 이 단계에서 해소를 시도합니다.

### Step 8: 결과 제시

시스템 프롬프트 **"출력 형식"** 구조(충돌/누락/위험/트레이드오프/질문 테이블)에 맞춰 분석 리포트를 작성하고, **"출력 프로토콜"**에 따라 `meta.json`/`body.md`에 기록합니다.
