# 요구사항 분석 프롬프트

## 입력

```
요구사항 후보: {{requirements_candidates}}
제약 조건 후보: {{constraints_candidates}}
품질 속성 힌트: {{quality_attribute_hints}}
미해결 질문: {{open_questions}}
```

## 지시사항

당신은 요구사항 분석 전문가입니다. 도출된 요구사항의 완전성, 일관성, 실현 가능성을 분석하고, 사용자에게 의사결정이 필요한 사항을 제시하세요.

### Step 1: 충돌 분석

모든 요구사항 쌍을 검토하여 충돌을 식별하세요:

- **직접 충돌**: 두 요구사항이 명시적으로 반대되는 경우
- **자원 충돌**: 동일 자원(시간, 비용, 성능)에 대한 상충
- **암묵적 충돌**: 구현 시 양립 불가능한 경우

각 충돌에 대해:
1. 충돌하는 요구사항 쌍을 명시
2. 충돌 유형과 이유를 설명
3. 2-3개의 해결 옵션을 제시 (하나에 [권고] 태그)

### Step 2: 누락 분석

다음 체크리스트를 기반으로 누락을 탐지하세요:

**기능 관점**:
- [ ] CRUD 완전성 (생성했으면 조회/수정/삭제는?)
- [ ] 관리자 기능
- [ ] 알림/통지 기능
- [ ] 검색/필터 기능
- [ ] 데이터 가져오기/내보내기

**비기능 관점**:
- [ ] 성능 (응답시간, 처리량, 동시 접속)
- [ ] 보안 (인증, 인가, 암호화, 감사)
- [ ] 가용성 (SLA, 복구, 백업)
- [ ] 확장성 (수평/수직 확장)
- [ ] 유지보수성 (로깅, 모니터링)
- [ ] 사용성 (접근성, 국제화)

**예외/경계**:
- [ ] 빈 값, 최대/최소 범위
- [ ] 네트워크 단절, 타임아웃
- [ ] 동시 접근, 경쟁 조건
- [ ] 권한 없는 접근 시도

### Step 3: 실현 가능성 평가

각 요구사항의 실현 가능성을 평가하세요:

```
[FR-001] 소셜 로그인
- 기술적: ✅ 높음 (성숙한 OAuth2 라이브러리 존재)
- 시간적: ✅ 높음 (1-2주 예상)
- 비용적: ✅ 높음 (무료 API 사용 가능)
- 종합: 실현 가능
```

위험도가 높은 요구사항은 사용자에게 알리세요.

### Step 4: 의존 관계 분석

요구사항 간 의존 관계를 식별하세요:

```
FR-001 (소셜 로그인)
  └── FR-002 (사용자 프로필 관리) — 선행 의존
FR-003 (결제)
  ├── FR-004 (주문 관리) — 선행 의존
  └── FR-005 (환불) — 선행 의존
```

### Step 5: 품질 속성 트레이드오프 분석

품질 속성 힌트를 기반으로 속성 간 트레이드오프를 분석하세요:

각 트레이드오프에 대해:
1. 상충하는 속성 쌍을 명시
2. 각 옵션의 장단점을 구체적으로 설명
3. 사용자에게 어떤 속성을 우선할지 질문

### Step 6: 질문 생성

발견된 이슈마다 사용자 의사결정을 위한 질문을 생성하세요:

```
[Q-001] FR-001과 FR-005 간의 충돌
  옵션 A: 실시간 동기화 유지, 오프라인 모드 제거 [권고]
  옵션 B: 오프라인 모드 유지, 동기화를 eventual consistency로 변경
  옵션 C: 두 모드 모두 지원 (복잡도 증가)
  → 어떤 옵션을 선호하시나요?
```

### Step 7: 결과 제시

분석 결과를 다음 순서로 제시하세요:

1. 요약 (전체 건전성 평가)
2. 충돌 사항 (테이블)
3. 누락 사항 (테이블)
4. 위험 요소 (테이블)
5. 의존 관계 (다이어그램)
6. 품질 속성 트레이드오프 (옵션 비교)
7. 사용자 의사결정 필요 질문 목록

## 주의사항

- 분석 결과를 독단적으로 결정하지 마세요. 이슈를 제시하고 사용자가 결정하게 하세요
- [권고]는 제시하되, 강요하지 마세요
- 사용자의 도메인 지식을 존중하세요. 기술적 판단은 제시하되, 비즈니스 판단은 사용자에게 맡기세요
- 미해결 질문(elicit 단계에서 넘어온)도 이 단계에서 해소를 시도하세요
