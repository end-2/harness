# 요구사항 도출 프롬프트

## 입력

```
사용자 요청: {{user_request}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 핵심 원칙을 따라 사용자 요청을 분석하고 대화를 통해 요구사항을 점진적으로 도출하세요.

### Step 1: 입력 구체성 판단

사용자 요청의 구체성 수준을 판단하고 시스템 프롬프트 **"적응적 질문 전략"** 의 고수준/중간 수준/상세 입력 중 해당하는 전략을 선택합니다.

### Step 2: 대화 상태 초기화

시스템 프롬프트 **"대화 상태 관리"** 형식(확인됨/미확인/가정됨)으로 상태를 추적하며 대화를 진행합니다.

### Step 3: 질문 생성

선택된 전략의 질문 예시를 참조하여 targeted question을 생성합니다. 한 번에 3개 이하의 질문을 제시하며, 우선순위는 다음과 같습니다:

1. 시스템의 핵심 목적과 범위
2. 핵심 기능 요구사항
3. 비기능 요구사항
4. 제약 조건
5. 경계 조건과 예외

### Step 4: 중간 확인

시스템 프롬프트 **"확인과 요약"** 형식에 따라 3-5턴마다 또는 주요 주제 전환 시 지금까지 이해한 내용을 요약하여 사용자 확인을 요청합니다.

### Step 5: 이해관계자 관점 질문

시스템 프롬프트 **"이해관계자 역할 대행"** 절의 관점(최종 사용자, 운영자, 보안 담당자, 규제 준수)에서 사용자가 고려하지 못한 질문을 제기합니다.

### Step 6: 종료 판단

시스템 프롬프트 **"대화 종료 조건"** 중 하나에 해당하면 도출을 마무리하고 "analyze 단계로 넘어가겠습니다"라고 안내합니다.

### Step 7: 산출물 정리

시스템 프롬프트 **"출력 형식"** (요구사항 후보/제약 조건 후보/품질 속성 힌트/미해결 질문) 구조로 산출물을 정리하고, **"출력 프로토콜"**에 따라 `meta.json`/`body.md`에 기록합니다.

## 주의사항

- 사용자의 답변을 왜곡하지 마세요. 사용자가 말한 그대로를 source에 기록합니다
- 가정을 세운 경우 반드시 [가정됨]으로 표시하고 사용자에게 확인을 요청하세요
- 사용자가 "모르겠다"고 답하면 합리적인 기본값을 제안하되, 사용자의 확인을 받으세요
- 기술적 결정(아키텍처, 기술 스택)은 이 단계에서 확정하지 않습니다. 제약 조건으로만 기록합니다
