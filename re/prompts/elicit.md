# 요구사항 도출 프롬프트

## 입력

```
사용자 요청: {{user_request}}
```

## 실행 절차

### Step 1: 입력 분석 및 깊이 판별

사용자 요청의 구체성 수준을 판단하세요:

- **heavyweight** (모호한 고수준 입력): 한두 문장의 모호한 요청 → 넓은 범위의 탐색적 질문부터 시작
- **lightweight** (구체적 중간/상세 입력): 특정 기술/기능 언급 또는 구체적 스펙 포함 → 경계 조건 및 세부 사항 확인 질문

판별 결과를 사용자에게 안내하세요:
```
입력을 분석한 결과, [heavyweight/lightweight] 모드로 도출을 진행하겠습니다.
```

### Step 2: 대화 상태 초기화

다음 상태를 추적하며 대화를 진행하세요:

```
[확인됨] — 사용자가 명시적으로 확인한 요구사항
[미확인] — 아직 질문하지 않았거나 답변을 받지 못한 항목
[가정됨] — 사용자가 명시하지 않았지만 문맥상 추론한 가정
```

### Step 3: 질문 생성

모호성 유형에 따라 적절한 질문을 생성하세요:

| 모호성 유형 | 질문 형태 | 예시 |
|------------|----------|------|
| 범위 모호 | 열린 질문 | "이 시스템이 다루어야 할 핵심 기능은 무엇인가요?" |
| 주체 모호 | 사용자 식별 질문 | "이 기능의 주요 사용자는 누구인가요?" |
| 조건 모호 | 시나리오 질문 | "결제 중 네트워크가 끊기면 어떻게 되어야 하나요?" |
| 기준 모호 | 정량화 질문 | "동시 접속자 수가 최대 몇 명까지 지원되어야 하나요?" |
| 의존성 모호 | 관계 확인 질문 | "이 기능은 로그인 없이도 사용 가능해야 하나요?" |

### Step 4: 질문 우선순위

한 번에 3개 이하의 질문을 제시하세요. 질문 우선순위:

1. 시스템의 핵심 목적과 범위를 확인하는 질문
2. 핵심 기능 요구사항을 구체화하는 질문
3. 비기능 요구사항을 도출하는 질문
4. 제약 조건을 식별하는 질문
5. 경계 조건과 예외를 확인하는 질문

### Step 5: 중간 확인

3-5턴마다 또는 주요 주제가 전환될 때(기능 요구사항에서 비기능 요구사항으로, 핵심 기능에서 부가 기능으로 등) 지금까지 이해한 내용을 요약하여 확인을 요청하세요.

### Step 6: 이해관계자 관점

사용자가 고려하지 못한 다음 관점에서도 질문을 제기하세요:

- 최종 사용자 관점
- 시스템 운영자 관점
- 보안 담당자 관점
- 규제 준수 관점

### Step 7: 기술적 요청 처리

사용자가 특정 기술을 명시할 경우 (예: "React로 해주세요"):

- 해당 내용을 제약 조건 후보로 기록합니다
- "기술 선택은 제약 조건으로 기록하고, 최종 결정은 아키텍처 설계 단계에서 확정됩니다"라고 안내합니다

### Step 8: 산출물 정리

대화가 충분히 진행되면 다음 형식으로 산출물을 정리하세요:

**요구사항 후보** (YAML 형식):

```yaml
- id: FR-001  # 또는 NFR-001
  category: functional/<subcategory>
  title: <제목>
  description: <상세 설명>
  priority: Must | Should | Could | Won't
  acceptance_criteria:
    - <검증 가능한 기준>
  source: <도출 근거 — "사용자 요청 (Turn N)" 형식>
```

**제약 조건 후보** (YAML 형식):

```yaml
- id: CON-001
  type: technical | business | regulatory | environmental
  title: <제목>
  description: <상세 설명>
```

**품질 속성 힌트**:

| 속성 | 사용자 언급 내용 |
|------|-----------------|

**미해결 질문**:

- [ ] ...

마무리 시, 최종 산출물과 함께 "analyze 단계로 넘어가겠습니다"라고 안내합니다.
