# 요구사항 리뷰 프롬프트

## 입력

```yaml
요구사항 명세: {{requirements_spec}}                   # YAML 배열
제외 항목: {{excluded_items}}                           # YAML 배열
제약 조건: {{constraints}}                              # YAML 배열
품질 속성 우선순위: {{quality_attribute_priorities}}     # YAML 배열
```

## 실행 절차

### Step 1: 요구사항 명세 검증

각 요구사항에 대해 다음을 검증하세요:

**SMART 검증**:
- Specific: 하나의 구체적 기능/속성만 기술하는가?
- Measurable: 정량적 acceptance_criteria가 포함되어 있는가?
- Achievable: 기술적으로 구현 가능한가?
- Relevant: 프로젝트 목표와 관련 있는가?
- Traceable: source가 명확하고 추적 가능한가?

**모호성 탐지** — 다음 패턴을 찾아 플래그하세요:
- "적절한", "합리적인", "충분한" → 정량화 필요
- "등", "기타", "필요에 따라" → 범위 명확화 필요
- "가능한 한", "최대한" → 목표치 필요
- "빠르게", "안전하게" → 메트릭 필요

**추적성 검증**:
- 모든 FR에 acceptance_criteria가 있는가?
- NFR이 관련 FR과 연결되어 있는가?
- dependencies가 유효한 ID를 참조하는가?
- 순환 의존이 없는가?

### Step 2: 제약 조건 검증

**상호 모순 탐지**:
- 제약 간 양립 불가능성 검사
- 비즈니스 제약과 기술 제약 간 충돌 검사

**정합성 확인**:
- 제약이 요구사항 구현을 불가능하게 만들지 않는가?
- rationale이 충분한가?
- flexibility 분류가 적절한가?

**누락 체크**:
- 기술 스택 제약 확인 여부
- 예산/비용 제약 확인 여부
- 법적/규제 제약 확인 여부
- 운영 환경 제약 확인 여부

### Step 3: 품질 속성 우선순위 검증

**metric 검증**:
- 각 metric이 실제로 측정 가능한가?
- 합격 예시: "P95 응답시간 < 200ms"
- 불합격 예시: "응답이 빨라야 함"

**트레이드오프 검증**:
- trade_off_notes가 실질적 트레이드오프를 설명하는가?
- 상위 속성이 하위 속성에 미치는 영향이 명시되어 있는가?

**일관성 검증**:
- NFR의 metric과 품질 속성의 metric이 일치하는가?
- 제약이 품질 속성 달성을 방해하지 않는가?

### Step 4: 제외 항목 검증

- Won't 항목의 제외 사유가 타당한가?
- reconsider_trigger가 명시되어 있는가?
- 제외된 항목이 포함된 항목과 의존 관계가 있지는 않은가?

### Step 5: 후속 스킬 소비 적합성 판정

각 소비자 스킬에 대해 판정하세요:

**arch:design**:
- [ ] 아키텍처 드라이버가 될 품질 속성에 명확한 metric이 있는가?
- [ ] 설계 제약(기술 스택, 환경)이 충분한가?
- [ ] 컴포넌트 분리에 필요한 기능 분류가 있는가?

**qa:strategy**:
- [ ] 모든 FR에 테스트 가능한 acceptance_criteria가 있는가?
- [ ] NFR에 측정 가능한 목표치가 있는가?
- [ ] 테스트 우선순위를 결정할 priority가 부여되어 있는가?

**impl:generate**:
- [ ] 구현 범위가 명확한가?
- [ ] 기술 제약이 구체적인가?
- [ ] 의존 관계가 구현 순서를 결정하기에 충분한가?

**sec:threat-model**:
- [ ] 보안 관련 NFR이 존재하는가?
- [ ] 규제 제약이 식별되어 있는가?
- [ ] 인증/인가 관련 요구사항이 있는가?

**devops:slo**:
- [ ] SLO로 변환 가능한 metric이 있는가?
- [ ] 가용성/성능 목표치가 명확한가?
- [ ] 배포 관련 제약이 있는가?

### Step 6: 이슈 분류 및 제시

발견된 이슈를 심각도별로 분류하세요:

- **Critical**: 후속 스킬 소비 불가 → 반드시 수정
- **Major**: 품질 저하 또는 모호성 → 수정 권고
- **Minor**: 개선 가능 → 선택적 개선
- **Info**: 참고 사항 → 정보 제공

### Step 7: 최종 판정

전체 명세에 대해 판정하세요:

- **APPROVED**: Critical/Major 이슈 없음, 후속 스킬 소비 가능
- **CONDITIONAL**: Minor 이슈만 존재, 후속 진행 가능하나 개선 권고
- **REJECTED**: Critical 이슈 미해소, spec 에이전트로 회귀하여 수정 후 재리뷰 필요

### Step 8: 사용자 에스컬레이션

리뷰어가 독단적으로 판단할 수 없는 이슈를 사용자에게 에스컬레이션하세요:

```
[ESC-001] <이슈 설명>
  배경: ...
  선택지: A / B / C
  리뷰어 의견: ...
  → 어떤 방향으로 진행할까요?
```
