# 요구사항 리뷰 프롬프트

## 입력

```
요구사항 명세: {{requirements_spec}}
제약 조건: {{constraints}}
품질 속성 우선순위: {{quality_attribute_priorities}}
```

## 지시사항

시스템 프롬프트에 정의된 역할과 규칙을 따라 세 섹션 명세 문서를 검증하고 후속 스킬 소비 적합성을 판정하세요.

### Step 1: 요구사항 명세 검증

시스템 프롬프트 **"요구사항 명세 검증"** 절을 적용합니다:

- **SMART 기준 검증**: SMART 표의 다섯 기준으로 각 요구사항 점검
- **모호성 탐지**: 모호 표현 패턴 목록으로 스캔
- **FR-NFR 추적성**: 모든 FR에 acceptance_criteria가 있는지, NFR이 관련 FR과 연결되어 있는지, dependencies가 유효한 ID를 참조하는지, 순환 의존이 없는지 확인

### Step 2: 제약 조건 검증

시스템 프롬프트 **"제약 조건 검증"** 절에 따라 제약 간 상호 모순, 요구사항과의 정합성, rationale·flexibility 적절성, 누락 제약 체크리스트를 검사합니다.

### Step 3: 품질 속성 우선순위 검증

시스템 프롬프트 **"품질 속성 우선순위 검증"** 절에 따라 각 metric이 측정 가능한지, trade_off_notes가 실질적인지, NFR metric과 품질 속성 metric이 일관적인지 확인합니다.

### Step 4: 후속 스킬 소비 적합성 판정

시스템 프롬프트 **"후속 스킬 소비 적합성 체크"** 표의 다섯 소비자 스킬(arch:design, qa:strategy, impl:generate, sec:threat-model, devops:slo)에 대해 각각 필요한 정보가 충분한지 판정합니다.

### Step 5: 이슈 분류

발견된 이슈를 시스템 프롬프트 **"이슈 분류"** 표의 심각도(Critical/Major/Minor/Info)로 분류합니다.

### Step 6: 최종 판정

시스템 프롬프트 **"최종 판정"** 기준에 따라 APPROVED/CONDITIONAL/REJECTED 중 하나를 결정합니다.

### Step 7: 사용자 에스컬레이션

리뷰어가 독단적으로 판단할 수 없는 이슈를 다음 형식으로 사용자에게 에스컬레이션합니다:

```
[ESC-001] <이슈 설명>
  배경: ...
  선택지: A / B / C
  리뷰어 의견: ...
  → 어떤 방향으로 진행할까요?
```

### Step 8: 결과 기록

시스템 프롬프트 **"출력 형식"** 구조로 리뷰 리포트를 작성하고, **"출력 프로토콜"**의 승인 판정 절차에 따라 `--verdict`와 `progress`를 갱신합니다.

## 주의사항

- 리뷰는 건설적이어야 합니다. 문제만 지적하지 말고 개선 방안을 함께 제시하세요
- 비즈니스 의사결정(우선순위, 범위)은 사용자에게 에스컬레이션하세요
- 형식적 완전성뿐 아니라 내용의 실질적 품질도 검증하세요
- 후속 스킬 소비 적합성은 반드시 검증하세요 — 이것이 이 리뷰의 핵심 목적입니다
